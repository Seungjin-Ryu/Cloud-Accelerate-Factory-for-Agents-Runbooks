# Phase 2: Execution & Validation — Delivery Guide

> **Phase Duration:** 2–4 weeks (3–6 week sprints, repeatable wave-based delivery)
> 
> **Owner:** Vendor Engineer + Vendor Architect, supported by Cloud Factory Technical Lead (CFTL)
>
> **Audience:** Vendor Engineer, Vendor Architect, CFTL, Customer POC

---

## Objective

Execute the approved solution design from Phase 1 by building, integrating, and validating the
agent end-to-end in the customer's environment.

This phase delivers a **fully functional, tested agent solution** — ready to hand over to the
customer for User Acceptance Testing (UAT) in Phase 3. Delivery follows a **repeatable weekly
wave model**, with each wave focused on incremental build, integration, and validation progress.

> ⚠️ **Do not begin Phase 2 without all Phase 1 Exit Criteria confirmed.**
> All environment access, licenses, scope, and architecture must be locked before execution starts.

---

## Entry Criteria

The following must be confirmed before Phase 2 begins:

- [ ] All Phase 1 Exit Criteria met and signed off (see [Phase 1 Delivery Guide](../Phase-1-Prerequisites-Scoping/Delivery-Guide.md))
- [ ] Execution plan (task list with dates, owners, milestones) approved by customer and CFTL
- [ ] Customer has provided hands-on-keyboard environment access — OR — session-sharing
  arrangement confirmed as fallback
- [ ] Draft solution architecture reviewed and approved in Phase 1
- [ ] Dev/Test environment provisioned and accessible to Vendor team
- [ ] Weekly checkpoint cadence is active (calendar invites sent, first session completed)
- [ ] Project tracker shared and accessible to all stakeholders

## Task List

### 2.1 Agent Build (Copilot Studio / Foundry)

The core delivery activity: build the agent solution according to the approved architecture.
Delivery is executed in **repeatable weekly waves**, with each wave producing a testable increment.

#### Copilot Studio Agent Build
- [ ] Create or configure the dedicated **Copilot Studio environment** (non-default, project-specific)
- [ ] Build agent topics, trigger phrases, and conversation flows aligned to the target scenario
- [ ] Configure **generative AI orchestration** (Generative Answers, GPT-based topic fallback, etc.)
- [ ] Set up **authentication** within Copilot Studio (Azure AD / Entra ID SSO, anonymous, or manual)
- [ ] Apply **LLM output steering** if required (prompt engineering, system prompts, tone/style tuning)
- [ ] Configure agent **publishing settings** (Teams, Copilot, web chat, custom channel)
- [ ] Register agent with **Agent 365** (required for all CAF deliveries)

#### Azure AI Foundry Build (if applicable)
- [ ] Provision Azure AI Foundry resources (Azure OpenAI, Azure AI Search, Document Intelligence)
- [ ] Implement the **RAG pipeline** (index creation, chunking strategy, embedding model selection)
- [ ] Configure **Azure AI Search** index with customer data sources (SharePoint, Blob, Dataverse, etc.)
- [ ] Implement document processing pipeline if Document Intelligence is in scope
- [ ] Deploy application on **Azure Functions or Azure Container Apps**
- [ ] Apply **Standard or ZTA architecture** based on data classification (scrubbed vs. confidential)
  - Standard Architecture: for non-confidential / scrubbed data environments
  - ZTA Architecture: for confidential / production data with strict access controls
- [ ] Use **Bicep/Terraform IaC templates** for environment configuration and reproducibility
- [ ] Configure Key Vault and Managed Identity integrations

> 💡 **Tip:** Leverage pre-built Microsoft reference architectures and IaC templates wherever
> possible to accelerate delivery. Custom code or deep re-architecture is out of CAF scope.

---

### 2.2 Integration Setup

Connect the agent to the required data sources, APIs, and external systems as defined in the
approved architecture.

#### Connectors
- [ ] Configure **Power Platform connectors** (standard or custom) for required data sources
- [ ] Set up **Microsoft Graph connectors** if SharePoint/Teams/OneDrive data access is needed
- [ ] Configure **Dataverse connectors** for CRM or ERP data integration
- [ ] Validate connector authentication and permission scopes (least-privilege principle)
- [ ] Test each connector end-to-end in the Dev environment before moving to Test

#### APIs
- [ ] Identify all **external APIs** required by the scenario (REST, OData, SOAP, MCP)
- [ ] Configure **Power Automate flows** to invoke APIs from Copilot Studio (action-based triggers)
- [ ] Implement **API authentication** (OAuth 2.0, API key, Managed Identity)
- [ ] Validate API response handling and error management within the agent flow
- [ ] Document all API endpoints, authentication methods, and data contracts

#### Data Sources
- [ ] Confirm all required **data sources are accessible** from the agent environment:
  - [ ] SharePoint / OneDrive (document libraries, lists)
  - [ ] Dataverse tables
  - [ ] Azure Blob Storage
  - [ ] SQL / PostgreSQL / Cosmos DB (if applicable)
  - [ ] Microsoft Fabric / OneLake (if applicable)
- [ ] Validate **data freshness and indexing** for AI Search (confirm index refresh schedule)
- [ ] Confirm **data classification** for each source (public, internal, confidential) and apply
  appropriate architecture (Standard vs. ZTA)

> ⚠️ **Out of Scope Reminder:** CRM integrations (e.g., Salesforce), third-party system
> integrations beyond standard connectors, and App DevOps pipelines are not in CAF scope.
> Escalate to CFTL if such requests arise during build.

### 2.3 Governance & Security

Ensure the agent is secure, observable, and governable before handover to Phase 3.

#### Agent Landing Zone Configuration (if required)
- [ ] Deploy **Agent Landing Zone** if foundational governance and environment setup is in scope:
  - [ ] Configure Power Platform **Data Loss Prevention (DLP) policies**
  - [ ] Set up **environment-level security groups** and role assignments
  - [ ] Configure **tenant-level agent management settings** (who can create/publish agents)
  - [ ] Enable **audit logging** for agent interactions and connector usage
- [ ] Apply **Responsible AI guardrails**:
  - [ ] Content moderation settings in Copilot Studio (topic filtering, safety prompts)
  - [ ] Configure fallback behaviors for out-of-scope or sensitive queries
  - [ ] Implement conversation logging for review and compliance
- [ ] Configure **observability and monitoring**:
  - [ ] Enable Copilot Studio conversation analytics (session volume, escalation rate, CSAT)
  - [ ] Set up Azure Monitor / Application Insights for Foundry-based agents
  - [ ] Define alerting thresholds for error rates and latency
- [ ] Review and confirm **authentication model** is correctly implemented:
  - [ ] Entra ID SSO configured and tested
  - [ ] Service account permissions scoped to minimum required access
  - [ ] No over-permissioned accounts or shared credentials

> 📋 **Reference:** Governance & Security patterns → [../../03-patterns/Agent-Landing-Zone.md](../../03-patterns/Agent-Landing-Zone.md)

### 2.4 Internal Validation

Validate the agent solution in Dev and Test environments before handover to the customer.

#### Development Environment Testing
- [ ] Conduct **unit-level testing** of individual topics, flows, and API integrations
- [ ] Validate **generative AI response quality**:
  - [ ] Test with representative user queries across key scenarios
  - [ ] Verify grounding (responses cite correct data sources, no hallucination)
  - [ ] Confirm tone, language, and format match customer requirements
- [ ] Test **authentication flows** end-to-end (login, token refresh, session timeout)
- [ ] Validate **connector and API responses** under normal and edge-case inputs
- [ ] Document all bugs and issues found in the project tracker with severity ratings

#### Test Environment Validation
- [ ] Promote build from Dev to **Test environment**
- [ ] Conduct **end-to-end functional testing** covering the full agent conversation flow
- [ ] Perform **regression testing** after each fix or change
- [ ] Test **multi-turn conversation** scenarios (context retention, slot-filling, disambiguation)
- [ ] Validate **fallback and error handling** behaviors (no-match, API timeout, data unavailable)
- [ ] Conduct **performance baseline testing** (response latency, throughput under expected load)
- [ ] Prepare **internal test report** summarizing test coverage, pass/fail rates, and open issues
- [ ] CFTL reviews and approves test results before customer handover

> 💡 **Note:** The Factory team conducts **rudimentary internal testing**. Full UAT is the
> customer's responsibility and is handled in Phase 3. CAF can provide guidance on evaluation
> frameworks and test scripts to help the customer prepare.

---

### 2.5 Weekly Checkpoint Execution

Maintain delivery momentum and stakeholder alignment through structured weekly reviews.

- [ ] Hold **weekly checkpoint meeting** with all stakeholders (CFTL, Vendor PM, Customer POC, CSA)
- [ ] Cover the following agenda at each checkpoint:
  - **Progress review:** tasks completed this week vs. plan
  - **Demo (if available):** show working increment to customer for early feedback
  - **Blockers:** surface any blockers and assign owners with resolution deadlines
  - **Risks:** flag any risks to timeline, scope, or quality
  - **Next week plan:** confirm tasks for the coming week and responsible parties
- [ ] Distribute **meeting notes and updated action tracker** within 24 hours of each checkpoint
- [ ] Escalate blockers immediately if they cannot be resolved within the current week
  (see Escalation Path below)
- [ ] Update the **project tracker** after each checkpoint to reflect current status

> 📊 **Weekly Wave Model:** Each week represents one delivery wave. The team aims to produce
> a **testable increment** at the end of each wave — progressing from initial build → integration
> → full end-to-end validation.

## Exit Criteria

All of the following must be confirmed before moving to **Phase 3: Customer Testing & Production Cutover**:

- [ ] ✅ Agent build complete and deployed in the **Test environment**
- [ ] ✅ All integrations (connectors, APIs, data sources) validated end-to-end
- [ ] ✅ Governance and security configuration applied (DLP, auth, audit logging)
- [ ] ✅ Agent Landing Zone deployed and confirmed (if in scope)
- [ ] ✅ Internal test report completed — no **critical or high-severity** open issues
- [ ] ✅ CFTL has reviewed and approved internal test results
- [ ] ✅ Agent registered with **Agent 365**
- [ ] ✅ Agent published to the correct channel(s) (Teams, Copilot, web) in the Test environment
- [ ] ✅ Customer notified and UAT preparation guide shared
- [ ] ✅ Phase 3 kickoff date confirmed with customer

## Known Issues & Mitigation

The following are common issues encountered during Phase 2 CAF engagements, along with
recommended mitigation strategies:

| # | Known Issue | Impact | Mitigation |
|---|---|---|---|
| 1 | **Environment access delayed post-Phase 1** | Blocks all build activities | Escalate immediately to CFTL → Account Team CSA. Consider session-sharing as interim fallback. |
| 2 | **Data source not accessible / indexing errors** | AI Search returns poor or empty results | Verify network rules, permissions, and index configuration. Confirm data is ingested and chunked correctly. |
| 3 | **Generative AI response quality issues** (hallucination, off-topic answers) | Poor agent accuracy; risk to customer trust | Refine system prompts, adjust grounding settings, review chunking strategy, add topic-level guardrails. |
| 4 | **Connector authentication failures** (OAuth token expiry, permission errors) | Integration broken; agent cannot retrieve data | Review app registration scopes, validate token refresh flows, ensure Managed Identity is correctly configured. |
| 5 | **Scope creep** (customer requests features beyond agreed scope) | Timeline risk; delivery quality impact | CFTL flags as out-of-scope at next checkpoint. Document in tracker. Route to Phase 3 or post-delivery if needed. |
| 6 | **API latency / timeout issues** | Poor user experience; agent appears unresponsive | Implement retry logic in Power Automate flows; set appropriate timeout thresholds; consider async patterns. |
| 7 | **DLP policy conflicts blocking connectors** | Integration blocked by Power Platform governance policies | Review DLP policy with customer IT Admin. Request policy exception or use approved connector alternatives. |
| 8 | **Agent Landing Zone not completed in Phase 1** | Governance gaps; potential security risk in production | Escalate to CFTL. Prioritize ALZ setup as first task in Phase 2 before proceeding with agent build. |
| 9 | **Vendor resource availability gap** (time zone, skill mismatch) | Delivery delay | CFTL requests regional resource reallocation from Factory Program Manager. Document impact in project tracker. |
| 10 | **Customer POC unavailable for validation** | Cannot confirm requirements or review demos | Identify backup POC. CFTL escalates to Executive Sponsor if sustained unavailability threatens timeline. |

## Escalation Path

| Issue Type | First Contact | Escalation |
|---|---|---|
| Build blocker (technical) | Vendor Architect → CFTL | CFTL → CAF Technical Lead (Regional CFTL) |
| Environment / access blocker | Vendor PM → Customer POC | CFTL → Account Team CSA |
| Scope dispute | CFTL assesses fit | CFTL documents decision; escalate to CAF PM if unresolved |
| Governance / security concern | CFTL → Customer IT Admin | Account Team CSA → Customer Executive Sponsor |
| Timeline at risk | CFTL flags in weekly checkpoint | CAF Program Manager notified |
| Vendor resource issue | CFTL → CAF Program Manager | CAF Program Manager reallocates resources |
