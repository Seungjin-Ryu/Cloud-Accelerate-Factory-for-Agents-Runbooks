# Phase 3: Customer Testing & Production Cutover — Checklist

> **Phase Duration:** 1–2 weeks  
> **Owner:** Customer POC (primary), supported by CFTL + Vendor Architect  
> **How to use:** Complete each section in order. UAT must be fully signed off before production cutover begins. All items must be checked (✅) before the CAF engagement is officially closed. Blockers must be logged in the project tracker immediately.

---

## 🧪 Section 1 — UAT (User Acceptance Testing)

### 1.1 UAT Preparation

- [ ] **UAT test users** identified and granted access to the Test environment
- [ ] **UAT test scenarios** defined by customer, covering:
  - [ ] Happy path (primary use cases)
  - [ ] Edge cases (unexpected or boundary inputs)
  - [ ] Error handling (API timeout, no-match, out-of-scope queries)
- [ ] **UAT success criteria** defined and agreed with CFTL:
  - [ ] Pass rate threshold confirmed (e.g., ≥ 90% of test cases pass)
  - [ ] Generative AI response quality criteria defined
    (e.g., groundedness, relevance, tone)
- [ ] **Defect logging process** confirmed (severity categories, tracker location, owners)
- [ ] UAT **start date, duration, and completion deadline** agreed with all stakeholders
- [ ] UAT guidance document shared with customer test team

### 1.2 UAT Execution

- [ ] Customer has executed **all defined UAT test scenarios**
- [ ] Defects identified during UAT logged in shared **UAT defect tracker** with:
  - [ ] Issue description
  - [ ] Severity (Critical / High / Medium / Low)
  - [ ] Status and resolution owner
- [ ] Factory team has triaged and resolved defects per agreed SLA:
  - [ ] Critical defects resolved same day ✅
  - [ ] High defects resolved within 1–2 business days ✅
  - [ ] Medium / Low defects resolved before cutover — OR — documented
    as known limitations ✅
- [ ] **Regression testing** completed after each defect fix
- [ ] Mid-UAT checkpoint meeting held to review progress and address blockers
- [ ] **No critical or high-severity open defects** remaining at UAT close

### 1.3 UAT Sign-off

- [ ] Customer POC confirms UAT **success criteria met**
- [ ] Customer POC provides formal **UAT sign-off** ✅
- [ ] UAT defect tracker marked as closed (all items resolved or accepted)
- [ ] CFTL confirms UAT results reviewed and production cutover approved

## 🚀 Section 2 — Production Cutover

### 2.1 Pre-Cutover Validation

- [ ] All UAT sign-off criteria confirmed met (Section 1.3 complete)
- [ ] **Production environment** provisioned and confirmed to match
  Test environment configuration:
  - [ ] Environment variables and connection references updated for production
  - [ ] Security groups and role assignments applied
  - [ ] DLP policies confirmed active in production environment
- [ ] **Production data sources** accessible and indexed correctly:
  - [ ] SharePoint / OneDrive
  - [ ] Dataverse / Azure AI Search
  - [ ] Other sources as applicable
- [ ] **Authentication** validated in production environment:
  - [ ] Entra ID SSO tested end-to-end
  - [ ] Service account permissions confirmed (least-privilege)
- [ ] **Monitoring and alerting** configured and active in production:
  - [ ] Copilot Studio analytics dashboard confirmed live
  - [ ] Azure Monitor / Application Insights active *(Foundry agents)*
  - [ ] Alerting thresholds validated
- [ ] **Cutover plan** reviewed and approved by customer POC and CFTL:
  - [ ] Cutover date and time confirmed
  - [ ] Responsible parties for each cutover step identified
  - [ ] Communication plan for end users confirmed

### 2.2 Rollback Plan

- [ ] **Rollback plan** documented and shared with customer IT Admin, covering:
  - [ ] Trigger conditions for rollback (what would cause a rollback decision)
  - [ ] Rollback steps (how to revert to Test environment or previous state)
  - [ ] Rollback owner and decision authority identified
  - [ ] Maximum acceptable downtime defined
- [ ] Customer IT Admin confirms rollback plan is understood and actionable
- [ ] Rollback plan stored in agreed handover location

### 2.3 Cutover Execution

- [ ] Agent build promoted from **Test → Production environment**
- [ ] Agent published to all agreed **production channels**:
  - [ ] Microsoft Teams
  - [ ] Microsoft 365 Copilot
  - [ ] Web chat / Custom channel *(if applicable)*
- [ ] **Agent 365 registration** updated to reflect production deployment
- [ ] **Final smoke test** executed in production:
  - [ ] Representative test queries validated across key scenarios
  - [ ] Authentication confirmed working for production users
  - [ ] Connectors and data sources responding correctly
  - [ ] Monitoring confirmed capturing live data
- [ ] **Pilot group access** confirmed (initial rollout group can access and use agent)

### 2.4 Go-Live Confirmation

- [ ] Customer POC approves **production go-live** ✅
- [ ] CFTL confirms **Factory delivery is complete** ✅
- [ ] Go-live **announcement sent to end users** by customer
- [ ] **Field CSA notified** for post-go-live production support continuity
- [ ] Go-live date and confirmation logged in project tracker

## 📚 Section 3 — Handover Documentation

### 3.1 Reference Architecture

- [ ] **Solution architecture document** (as-built) finalized and delivered, including:
  - [ ] Updated logical architecture diagram
  - [ ] Component descriptions (Copilot Studio, Foundry, connectors, data sources)
  - [ ] Authentication and identity model
  - [ ] Governance and security layer (DLP, Agent Landing Zone)
  - [ ] Integration map (all connectors, APIs, data sources with endpoints)
- [ ] Architecture document reviewed and approved by CFTL
- [ ] Stored in agreed handover location (SharePoint / Teams / GitHub)

### 3.2 Infrastructure as Code (IaC) Templates

- [ ] All **Bicep / Terraform IaC templates** delivered:
  - [ ] Azure AI Foundry resource provisioning templates *(if applicable)*
  - [ ] Azure AI Search index configuration scripts *(if applicable)*
  - [ ] Key Vault and Managed Identity setup templates *(if applicable)*
  - [ ] Agent Landing Zone deployment templates *(if applicable)*
- [ ] Templates include **inline comments** explaining each configuration parameter
- [ ] **README file** included with instructions on how to use each template
- [ ] Templates confirmed tested and reproducible

### 3.3 Training Materials

- [ ] **End user training** delivered and materials handed over:
  - [ ] How to access and interact with the agent
  - [ ] Key use cases and example prompts
  - [ ] How to provide feedback / report issues
- [ ] **Admin / operator training** delivered and materials handed over:
  - [ ] Copilot Studio environment management
  - [ ] DLP policy and governance management
  - [ ] Agent 365 monitoring and governance
  - [ ] Connector and integration maintenance
- [ ] **Maker / developer training** delivered and materials handed over *(if applicable)*:
  - [ ] Adding and editing topics and conversation flows
  - [ ] Updating knowledge base / AI Search index
  - [ ] Publishing agent updates
- [ ] Training completion confirmed by customer POC

### 3.4 Post-Deployment Guide

- [ ] **Post-deployment operations guide** delivered, covering:
  - [ ] Day-to-day monitoring (analytics review, alert response)
  - [ ] Knowledge base / AI Search index refresh process
  - [ ] Topic and flow update process
  - [ ] User management (access, permissions, security groups)
  - [ ] Troubleshooting guide (common issues and resolution steps)
  - [ ] Escalation contacts (CAF Central, Account Team CSA, Microsoft Support)
  - [ ] Next steps / expansion roadmap suggestions
- [ ] Post-deployment guide reviewed by customer POC and confirmed complete

### 3.5 Handover Package Completion

- [ ] All documents consolidated in a **single handover package**
- [ ] Handover package stored in agreed location accessible to customer
- [ ] Customer POC confirms receipt and completeness of handover package ✅

## 🎓 Section 4 — Customer Enablement

- [ ] **Change management owner** identified on the customer side
- [ ] Go-live **communication template** provided to customer and sent to end users
- [ ] **Feedback and support channel** set up and communicated to end users
- [ ] Customer has confirmed plan for **ongoing agent maintenance**:
  - [ ] Knowledge base refresh cadence defined
  - [ ] Topic update ownership assigned internally
  - [ ] Governance review cycle scheduled
- [ ] Expansion roadmap discussed with customer
  (next agent scenarios, additional user groups, new integrations)
